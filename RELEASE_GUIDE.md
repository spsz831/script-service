# RELEASE_GUIDE

`script-service` 的仓库级维护说明，主要面向这 3 类操作：

1. 发布或整理一个已有脚本
2. 更新 README 预览图
3. 新增一个新的脚本子项目

## 适用范围

| 项目 | 说明 |
|---|---|
| 主要平台 | Windows |
| 主要环境 | PowerShell |
| 当前仓库结构 | 一个目录对应一个相对独立的脚本项目 |
| 当前目标 | 保持每个脚本都能单独理解、单独复制、单独运行 |

## 一次发布前要检查什么

| 检查项 | 要求 |
|---|---|
| 根 README | 项目清单、状态、依赖、预览图是否与当前仓库一致 |
| 子项目 README | 参数、入口、风险边界、依赖说明是否过时 |
| 版本号 | 如果脚本已对外发布，确认对应 `VERSION` 是否需要更新 |
| 变更记录 | 如果本次改动值得记录，补到对应 `CHANGELOG.md` |
| 可运行性 | 至少做一次最小可行验证，不只看脚本语法 |
| 预览图 | 如果 UI、输出样式、菜单、终端结果变化明显，重新生成或更新截图 |
| 无关文件 | 不把临时文件、测试垃圾、个人日志误提交到仓库 |

## 发布前推荐流程

| 步骤 | 建议动作 |
|---|---|
| 1 | 先在根目录执行 `git status --short`，确认改动范围 |
| 2 | 逐个检查受影响脚本目录的 `README.md`、`VERSION`、`CHANGELOG.md` |
| 3 | 运行最小验证命令，确认脚本实际可用 |
| 4 | 如有展示改动，更新 `docs/previews` 或子项目 `docs/` 中的图片 |
| 5 | 只提交与本次发布相关的文件，保持提交粒度清晰 |
| 6 | 推送后到 GitHub 页面检查 README 渲染和图片显示是否正常 |

## 如何更新预览图

当前仓库已经提供一个统一生成脚本：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\generate-readme-previews.ps1
```

## 预览图输出位置

| 路径 | 作用 |
|---|---|
| `docs/previews/` | 根 README 使用的总览预览图 |
| `codex-windows-repair/docs/` | `codex-windows-repair` 项目内展示图 |
| `html-to-png/docs/` | `html-to-png` 项目内展示图 |
| `html-to-pdf/docs/` | `html-to-pdf` 项目内展示图 |
| `project-cleaner/docs/` | `project-cleaner` 项目内展示图 |

## 更新预览图时要注意什么

| 项目 | 说明 |
|---|---|
| 先生成再提交 | 不要只改 README 引用，忘记提交图片文件 |
| 展示目标 | 优先展示“这个脚本做成了什么”，而不是只展示命令本身 |
| 图片尺寸 | 根 README 预览图应尽量保持同一视觉比例 |
| 内容选择 | 优先展示最有辨识度的脚本，不必所有脚本都先补图 |
| 风险控制 | 图片里不要带个人路径、敏感目录、账号信息、密钥等内容 |

## 如何新增一个脚本子项目

推荐先复制当前仓库已经稳定的目录结构思路，而不是临时自由发挥。

## 最小目录结构

| 文件 / 目录 | 作用 |
|---|---|
| `README.md` | 说明脚本用途、参数、依赖、风险边界 |
| `VERSION` | 当前版本号 |
| `CHANGELOG.md` | 版本变更记录 |
| `LICENSE` | 许可证 |
| `docs/` | 预览图、示意图等文档素材 |
| 主脚本 | 例如 `.ps1` |
| 双击入口 | 例如 `.cmd`，如有必要 |

## 新增脚本推荐步骤

| 步骤 | 建议动作 |
|---|---|
| 1 | 先确定这个脚本解决的是单一问题，而不是混装多个用途 |
| 2 | 新建独立目录，例如 `pdf-tools/`、`network-tools/` 风格的单项目目录 |
| 3 | 先把主脚本跑通，再补 README、VERSION、CHANGELOG、LICENSE |
| 4 | 如果适合双击运行，再补 `.cmd` 启动器 |
| 5 | 至少做一次最小验证，并把验证过的行为写进 README |
| 6 | 如果脚本适合展示，再决定是否补预览图 |
| 7 | 最后把根 README 的项目清单、分类表、推荐路径、依赖表同步更新 |

## 新脚本加入根 README 时要同步哪些位置

| 位置 | 要不要更新 |
|---|---|
| `项目清单` | 必须 |
| `脚本分类总表` | 通常需要 |
| `推荐使用路径` | 如果是常用脚本，建议更新 |
| `状态 / 依赖` | 必须反映真实情况 |
| `预览图` | 只有具备代表性时再补 |

## 版本和变更记录建议

| 文件 | 建议 |
|---|---|
| `VERSION` | 保持简单，使用 `1.0.0` 这类可读版本号 |
| `CHANGELOG.md` | 只记录对用户有意义的变化，不写纯内部噪音 |
| 提交信息 | 尽量让 `feat`、`fix`、`docs` 区分清楚 |

## 提交前的最后检查

```powershell
git status --short
git diff --stat
```

如果要补生成后的预览图，也建议再跑一次：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\generate-readme-previews.ps1
```

## 当前仓库的实际维护原则

| 原则 | 说明 |
|---|---|
| 先可用 | 先保证脚本真的能用，再谈展示 |
| 先简单 | 优先低耦合、低依赖、可维护的实现 |
| 先验证 | 没验证过的行为，不要在 README 里写成已完成 |
| 先收口 | 一个提交尽量只做一类事情：脚本逻辑、文档、预览图分开更清楚 |
